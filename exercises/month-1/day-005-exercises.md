## 🎧 LISTENING TEXT — Day 5

### Listening 5.1: Git for Beginners (Chậm)

```
Git is a version control system. It helps developers track changes to their code. Let me explain how it works.

Imagine you are writing a document. Every time you make a change, Git takes a snapshot of your file. This snapshot is called a commit. Each commit has a unique number and a message that describes what you changed.

If you make a mistake, you can go back to a previous commit. This is very useful when something breaks in your code.

Git also lets you create branches. A branch is a separate copy of your code where you can work on a new feature without affecting the main code. When your feature is ready, you merge the branch back into the main code.

Many developers use GitHub to store their code online. GitHub is a website where you can push your code and share it with your team. Your teammates can pull your changes, review your code, and merge it.

If two people change the same part of a file, Git will report a conflict. You need to resolve the conflict by deciding which changes to keep.
```

### Listening 5.2: A Day Using Git (Bình thường)

```
Let me walk you through how I use Git in my daily work.

Every morning, I start by pulling the latest code from the main branch. I type "git pull origin main" in the terminal. This makes sure I have the most recent changes from my team.

Then, I create a new branch for the task I'm working on. I type "git checkout minus b feature slash login page." This creates a new branch called "feature slash login page."

I start coding. When I finish a piece of work, I stage my changes with "git add dot" and then commit them with "git commit minus m" followed by a description of what I did. For example, I might write "Add login form component."

I make several commits throughout the day. When I'm done with the feature, I push my branch to GitHub with "git push origin" followed by my branch name.

After that, I go to GitHub and create a pull request. I describe what I changed and ask a teammate to review my code. If they have suggestions, I make the changes and push again. Once the code is approved, I merge the pull request.

Sometimes I have a conflict when merging. This means someone else changed the same file. I open the file, look at both versions, and decide which code to keep. Then I commit the resolved conflict and finish the merge.
```

### Listening 5.3: Dictation

```
Clone the repository from GitHub.
Create a new branch for your feature.
Commit your changes with a clear message.
Push your code to the remote repository.
Open a pull request for code review.
Resolve the merge conflict carefully.
Pull the latest changes before you start working.
Use git stash to save your work temporarily.
```
